AWS with Terraform: why is my resource updated on every run?

AWS and Terraform'ing for noobs

April 18, 2020 Marcelo Andrade

4 minute read

  • Twitter

  • Google+

  • Facebook

  • Reddit

  • LinkedIn

  • StumbleUpon

  • Pinterest

  • Email

Note for Pros: Terraform noob tip. It’s not that relevant.

Preamble

I had a turbulent mission: learn both Terraform and AWS from scratch without ever using it before, building a full EKS cluster as IaaC. I also had two weeks.

So, as expected, it was a hurricane of copy and paste from examples in the Internet - mainly official Terraform documentation - and methodology was “if it’s created, it’s fine!".

A month after, it was time to actually read what was happening. So you start noticing where you failed without even knowing,

The matter at hand

On every plan/apply, my private subnet’s route table was getting updated. But I wasn’t changing anything!

So, whats wrong?

Well, the answer is pretty obvious in the output:

# module.aws-eks.aws_route_table.private will be updated in-place
~ resource "aws_route_table" "private" {
        id              = "rtb-0e22a048b2cfa002a"
        owner_id        = "149838453533"
        propagating_vgws = []
      ~ route            = [
          - {
              - cidr_block                = "0.0.0.0/0"
              - egress_only_gateway_id    = ""
              - gateway_id                = ""
              - instance_id              = ""
              - ipv6_cidr_block          = ""
              - nat_gateway_id            = "nat-009ca190cf91f7a97"
              - network_interface_id      = ""
              - transit_gateway_id        = ""
              - vpc_peering_connection_id = ""
            },
          + {
              + cidr_block                = "0.0.0.0/0"
              + egress_only_gateway_id    = ""
              + gateway_id                = "nat-009ca190cf91f7a97"
              + instance_id              = ""
              + ipv6_cidr_block          = ""
              + nat_gateway_id            = ""
              + network_interface_id      = ""
              + transit_gateway_id        = ""
              + vpc_peering_connection_id = ""
            },
        tags            = {
            "Name"                      = "tfes-eks/tfes-a01/RouteTable/Private"
            "devsres.com/author"        = "marcelo at
devsres.com"
            "devsres.com/project"      = "aws-eks"
            "devsres.com/cluster"      = "tfes-a01"
            "devsres.com/scid"          = "10316"
        }
        vpc_id          = "vpc-023ab8237b2fb8714"
    }

But since I was a noob, it wasn’t pretty obvious to me.

Lets try to read it again:

~ route            = [
          - {
              - cidr_block                = "0.0.0.0/0"
              - egress_only_gateway_id    = ""
              - gateway_id                = ""
              - instance_id              = ""
              - ipv6_cidr_block          = ""
              - nat_gateway_id            = "nat-009ca190cf91f7a97"
              - network_interface_id      = ""
              - transit_gateway_id        = ""
              - vpc_peering_connection_id = ""
            },
          + {
              + cidr_block                = "0.0.0.0/0"
              + egress_only_gateway_id    = ""
              + gateway_id                = "nat-009ca190cf91f7a97"
              + instance_id              = ""
              + ipv6_cidr_block          = ""
              + nat_gateway_id            = ""
              + network_interface_id      = ""
              + transit_gateway_id        = ""
              + vpc_peering_connection_id = ""
            },

Wait, maybe now I see:

# What exists now there:
...
              - gateway_id                = ""
              - nat_gateway_id            = "nat-009ca190cf91f7a97"
...

# What will replace it:
...
              + gateway_id                = "nat-009ca190cf91f7a97"
              + nat_gateway_id            = ""
...

Now I get it. The existent entry for a Nat Gateway on the route table rtb-0e22a048b2cfa002a will be replaced for an empty entry, while the field gateway_id will receive its value.

So it seems I screw up the route table entry resouce. Let’s revisit:

# Public route table: route for IGW
resource "aws_route_table" "public" {
     vpc_id = aws_vpc.main.id
     
     # TODO: remove the inline route creation, I know it will cause me problems in the future.
     route {
         cidr_block = "0.0.0.0/0"
         gateway_id = aws_internet_gateway.igw.id
     }
     tags = merge( map("Name", "${local.tag_name_prefix}/RouteTable/Public"),
                             local.tags_default
             )
}

resource "aws_route_table" "private" {
     vpc_id = aws_vpc.main.id

     # TODO: remove the inline route creation, I know it will cause me problems in the future.
    route {
         cidr_block = "0.0.0.0/0"
        gateway_id = aws_nat_gateway.ngw.id
     }
     tags = merge( map("Name", "${local.tag_name_prefix}/RouteTable/Private"),
                             local.tags_default
             )
}

So it seems I got Copy’nPaste’d here, and used gateway_id as the attribute to go for both IGW and NAT Gateway entries. I should replace the gateway_id on the aws_route_table.private for nat_gateway_id and everything will be fine from now on.

Lessons learned

  1. RTFM instead of just copying examples.

It’s even written there in a different background! I blame the pastel colors for skipping it.

NOTE on gateway_id and nat_gateway_id: The AWS API is very forgiving with these two attributes and the aws_route_table resource can be created with a NAT ID specified as a Gateway ID attribute. This will lead to a permanent diff between your configuration and statefile, as the API returns the correct parameters in the returned route table. If you’re experiencing constant diffs in your aws_route_table resources, the first thing to check is whether or not you’re specifying a NAT ID instead of a Gateway ID, or vice-versa. (Terraform on Route Tables)

  1. Pay very good attention on your Terraform plan outputs.

If you plan to XGH your IaaC like I’m doing, you better learn quick enough that “if it compiles it works” is still not true in this new world. Even if the Terraform apply doesn’t fail, it doesn’t mean it worked as intended.

  1. If you fail on #1 (RTFM), search for issues before you ask

It will help you to not feel embarrassed on Github when opening an Issue (maybe someone had this problem before?)
