AWS S3: Hosting an static Website

AWS S3 as a static website host

November 10, 2019 Marcelo Andrade

5 minute read

  • Twitter

  • Google+

  • Facebook

  • Reddit

  • LinkedIn

  • StumbleUpon

  • Pinterest

  • Email

Task at hand: host a static website on a S3 bucket. URL for the website will be: yellatcloud.devsres.com

Resources used: AWS S3, custom domain registered.

Two of the most repeated sentences on the Linkedin Learning AWS Courses I’ve met are:

  • AWS S3 (the object storage service) - is the most widely used Amazon service ever;
  • One can host static websites on AWS S3 trivially.

Is that so? Let’s check. Let’s host this pretty Hugo Website made just for this task.

Creating the AWS S3 bucket

Basically the most important thing here is: the name of the S3 bucket must match the URL of your website.

As you can’t rename a bucket, I’m guessing this information might save your time.

So, let’s go to AWS S3 and create the yellatcloud.desvres.com bucket. Don’t tell your troll friends about it, or they might create a bucket with the exact same name and then you’re out of luck, as they’re globally unique!

So, on the first screen, we pick:

  • Bucket name: yellatcloud.devsres.com
  • Region: whatever rows your boat.

On the second one:

  • Versioning: blank, let’s use Github to version it anyway;
  • Server access logging: meh, whatever, leaving it unchecked;
  • Tags: be creative here. every single AWS course also tells to put tags everywhere, so do it.
  • Object-level logging: no, thank you.
  • Default encryption: not that useful for a public website anyways;
  • Advanced settings:
    • Permanently allow objects in this bucket to be locked: now this is an interesting feature, but not for today’s practise. Let’s write it down to check it on the future!
  • Cloud Watch request metrics: no additional costs, please.

The third screen is about blocking public access. That’s exactly the opposite of what we want to do, so let’s uncheck everything.

  • Block all public access: uncheck.
  • Manage system permissions: Do not grant Amazon S3 Log Delivery group write access to this bucket. Indeed.

Fourth screen is just a review, and as we’re anxious to make it work, let’s just hit Create bucket without reading anything, as a good sysop.

Uploading the website to the S3 bucket

So I should be doing this the smart way (using the AWS cli), but let’s forget it for a minute and just try the awesomely beautiful web interface. So we click on the bucket name, and hit Get Started.

The first screen tells you can just drag and drop files and folders there. It has a button, Add files, let’s try it!

Eh, didn’t work for some reason. Wonder why.

So let’s try the drag and drop feature! it works, yay!

On the second screen, we have a gotcha that might catch you if you skip it without reading:

  • Manage users: nothing to do here, unless you’re dealing with many users on your account;
  • Access for other AWS accounts: similar to above, nothing to do here;
  • Manage public permissions: so, if you thought you had done enough by unmarking Block all public access, nah. On that screen, all it made was so it was possible to make stuff public, not that all stuff is public. Have to change it here to Grant public read access to this object(s).

The third screen is an interesting one: it’s where you choose the Storage Class of your files. i

  • Standart : it’s important to know all these options, as it’s what determines how much $$$ you’ll be flushing with the service. As this is a website, all files are equally relevant, so it makes no sense to pick any of the Glacier options (which are directed to backup/cold data). The other ones, like Intelligent Tiering, Standard-IA, One Zone-IA and Reduced Redundancy are oriented on providing options of different pricing. As this is not a S3 full guide, let’s pick Standart for now.

If you get too excited about clicking Next, you’ll miss a couple of options:

  • Encryption: three options here, let’s pick None and revisit this on a S3 specific article;
  • Metadata: you can add some specific headers (like Cache-Control or Content-Type) to the files you’re uploading to S3! Now that’s a nice thing. Just beware: you can’t change them later!
  • Tag: objects inside the bucket can have tags too! Now that’s nice (but useless for this example, so ignore).

Next screen is the review, so just give a glance and hit Upload.

Enabling the website for serving static content

So this is the step we’ve been waiting for so long. It’s hidden under the bucket’s Properties tab (maybe add a tab called Special Features? seems more markeatable!).

A lot of interesting properties here to look up after, like Object Lock and Requester pays (!). But we’re here for Static website hosting, the third box. Let’s click on it.

After the click, it grows and asks some more stuff:

  • Use this bucket to host a website: yeah, that’s the one. Let’s click.
  • Redirect requests: let’s see this one later.

It asks some more stuff, like Index document and Error document. Funny thing they’re required ones to save, so let’s write with the default values suggested and Save.

So it gave me my endpoint: http://yellatcloud.devsres.com.s3-website-sa-east-1.amazonaws.com . Everything should be ok, so if I hit the URL, I should just see the website, right?

Well, indeed it worked.

But that’s not the only way to do it. Most people will say that setting up a explicit bucket policy that allows public access is the right way to do it.

But since it’s too complicated, I made it on its own post.

But what about HTTPS?

Sorry, it’ll be a story for another time!

Key notes

  • You can’t rename S3 buckets in AWS;
  • S3 bucket names have to be globally unique - they’re not restricted to your account or region. Hope you’re not a fan of short names!
  • S3 bucket name must match the custom domain of the website or it will just not work.
  • S3 bucket names have to be DNS compliant, so the weirdest character you’ll be able to use is ‘-'.
  • S3 supports WORM!
  • S3 has a lot of storage classes you should check out.
